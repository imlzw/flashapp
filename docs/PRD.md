# 📄 “闪创” (FlashApp) 产品需求文档 (PRD)

## 1. 产品定义
“闪创”是一个基于 AI 智能体驱动的 H5 快速开发与托管平台。用户通过对话描述需求，系统自动完成：**代码生成 -> 实时预览 -> 自动部署 -> 迭代修改**。

## 2. 用户核心路径
1.  **输入需求**：用户在聊天框描述应用想法。
2.  **流式生成**：右侧模拟器通过 `iframe` 实时显示 AI 编写的代码。
3.  **自动部署**：代码生成结束，系统自动落盘，实现一键预览。
4.  **社区发现**：用户可将应用发布至“广场”，其他用户可进行“运行”与“重混”。
5.  **韧性编辑**：通过“重混”机制，即便是历史会话丢失的发布版本也能一键找回并继续迭代。

# 🛠️ 详细设计文件 (Technical Design)

## 1. 系统架构与技术栈
* **后端**: Go (Golang) —— 负责核心逻辑、API 中转、JSON 文件存储（计划迁移 SQLite）。
* **存储**: 
    * **应用元数据**: 当前采用 JSON 存储 (`store.go`)，支持用户关联与版本管理。
    * **应用代码**: 本地目录静态存储。
* **前端**: 原生 HTML/JS/CSS (Vanilla JS)。
* **部署**: 嵌入式可执行文件，单端口（默认 `:18080`）处理 API 与静态资源。

## 2. 存储设计 (极致轻量化)

### 2.1 数据库逻辑 (当前实现)
* **Users**: 存储 ID、用户名、加密 Hash。
* **Apps (私有)**: 存储应用 ID、标题、Prompt 记录、预览地址。
* **Published Apps (广场)**: 存储发布 ID、原始 App 关联、作者名、截图 URL、发布时间、公开状态。

### 2.2 文件存储结构
```text
apps/
└── {short_id}/
    ├── index.html   # 最新生成的 HTML 代码
    └── app.json     # 该应用的元数据快照
```

---

## 3. 核心功能流程设计

### 3.1 沉浸式交互逻辑 (UI/UX)
1.  **页眉状态管理**：
    *   在“广场”与“我的应用”列表页显示页眉。
    *   在“创作（聊天）”界面自动隐藏页眉，为手机壳预览提供全高展示空间。
    *   当进入“全屏预览”或“弹窗预览”时，页眉通过 `translateY` 动效优雅退场。
2.  **卡片化操作**：
    *   纯图标动作按钮（Play, Remix, Edit, Globe/Lock, Delete）。
    *   智能编辑：点击编辑时若原始 Session 丢失，自动触发 Fork 逻辑，实现“逻辑续传”。

### 3.2 智能体工作流 (Agent Workflow)
后端通过一个统一的 `Agent` 模块处理请求：
1.  **上下文加载**：若是修改请求，从本地 `apps/{id}/index.html` 读取旧代码。
2.  **流式透传**：调用 LLM API，将代码流实时推送至前端。

## 4. 关键接口定义 (Go API)

| 接口 | 方法 | 说明 |
| :--- | :--- | :--- |
| `/api/auth` | POST | 极简 JWT 登录/注册。 |

## 5. Nginx 泛域名映射配置
这是实现“一键预览”的核心：
```nginx
server {
    listen 80;
    server_name ~^(?<appid>.+)\.yourdomain\.com$;

    location / {
        root /var/www/flashapps/$appid;
        index index.html;
        # 针对 128MB 环境，关闭不必要的缓存，开启 gzip 压缩
        gzip on;
        add_header Cache-Control "no-store, no-cache, must-revalidate";
    }
}
```

## 6. 极致优化指南 (Survival Tips)

* **内存压缩 (Go)**：在 `main` 函数启动时设置环境变量 `GOMEMLIMIT=80MiB`。
* **SQLite 驱动**：使用 `modernc.org/sqlite`。它是纯 Go 实现，不需要 CGO，内存开销比 cgo 版本更可控。
* **前端渲染**：禁止前端存储过大的历史记录，每次修改后清空旧缓存。
* **日志控制**：关闭详细访问日志，仅保留 Error Log，防止磁盘 IO 撑爆 CPU。

## 7. 总结
本方案将服务器角色降到了最低：**它不运行 AI，不构建代码，不运行重量级数据库**。它仅仅是 API 的**搬运工**和文件的**托管员**。这种设计确保了即便在 128MB 的“古董级”服务器上，也能流畅支撑一个现代化的 AI 开发平台。
